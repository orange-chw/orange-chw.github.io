##分析
这里先介绍下load的方法的执行方式。load方法是NSObject的方法，NSObject则是oc里大部分类的基类。只要类被添加到runtime中，load的方法均会被调用，而且只会自动被调用一次（除非你自行有调用了次方法，不过一般不会这样做）。其调用方式是执行顺序是 类 -> 子类 ->分类 ，例如A继承与B会先执行B的load再执行A的load，如果分类实现了load方法也是一样会先执行类的load再执行此类的分类里的load方法。因此，此方法是不会被覆盖。

MJRefresh为了替换oc类tableview和collectionview的reloadData以达到调用其自定义的mj_reloadDataBlock，因此实现了分别定义了类别UITableView(MJRefresh)和UICollectionView(MJRefresh)并重写了类方法load。在load方法里MJRefresh做了一件事情就是使用runtime里的method_exchangeImplementations方法来实现用mj_reloadData替换tableview和collectionview里的reloadData方法。
在mj_reloadData的方法里我们看到其为了该方法能既能执行原有的reloadData方法又能执行mj_reloadDataBlock，他先调用了[self mj_reloadData]然后有调用了mj_reloadDataBlock。由于方法被交换过，因此此处的[self mj_reloadData]会调用reloadData方法并不会造成死循环，同时也达到了其目的调用原有的reloadData方法。

但是这里我也发现了一个问题，既然方法交换了，如果你调用了[self mj_reloadData]方法则会变成调用reloadData方法，那样岂不是会达到效果？在这里MJRefresh为了不让你这样做，则是将mj_reloadData做为私有方法，没有公开API，你是无法直接调用到的（当然你想调用还是调用的到的，这里就不多做讨论）。

##总结
从此处设计，能看出来method_exchangeImplementations的runtime方法的使用场景以及如何运用的思路。在此做了分析记录，也为以后如果碰到一些类似场景提供一些新的解决思路和方式。

iOS 关于app后台任务的问题

发表于 2017-12-18

##问题
最近在开发中有个需求，需要app退到后台继续执行每间隔60s调用心跳接口的任务。在此，不讨论为何如此设计，只论实现。

app前台运行实现方案

程序运行时，只需要运用NSTimer类即可实现该功能。
具体操作如下

_timer = [NSTimer scheduledTimerWithTimeInterval:60.0 
                                          target:self 
                                            selector:@selector(heartBeatClick)
                                            userInfo:nil
                                             repeats:YES];

了解NSTimer使用方法点击这里

app后台运行实现方案

首先，来了解下app的后台运行的机制。

在开发中如果没有进行特殊的设置，app在进入后台后，系统会将分配给app的资源进行回收，此时会暂停所有正在执行的计时器以及正在执行的业务逻辑，app会处于挂起状态，再过一会app就会被kill。

由于我们在接单状态下会使用持续定位功能。因此在此模块，我们运用持续定位的backgorund模式来实现了app长期处于backgound状态。（此方法在ios9.0以上可用）

1.先配置项目target的backgound模式

将Location updates和 background fetch选项打勾

2.由于项目使用的是高德地图，在此用高德地图为代码示例。我们此处指使用定位，未使用地图，因此使用的是高德的AMapLocationManager类。示例代码如下

_locationManager = [[AMapLocationManager alloc]init];
_locationManager.desiredAccuracy = kCLLocationAccuracyBest;
_locationManager.distanceFilter = 30;
_locationManager.delegate = self;
_locationManager.locatingWithReGeocode = YES;
_locationManager.pausesLocationUpdatesAutomatically = NO;
//此处的关键部分在于_locationManager要打开允许后台刷新定位
//此字段是在iOS 9.0开放的，因此要在此加上版本判定
if ([[UIDevice currentDevice].systemVersion floatValue] > 9.0) {
    _locationManager.allowsBackgroundLocationUpdates = YES;
}

3.通过_locationManager开启持续定位

[_locationManager  startUpdatingLocation]

之后，在持续定位模式下面，app在进入后台后会保持background模式，计时器会继续运行不会被暂停中断。

此方法的缺点是耗电，app在开启持续定位手机的耗电会比较快。但是项目的接单设计必须要用户持续定位，以方便用户能在用户端看到技术员的最新位置，因此此处正好也符合了app后台间隔1分钟持续调用心跳包的需求。

总结

app的后台模式不仅使用后台定位能实现，还有一系列系统级别的后台功能可以使app退到后台后变成background模式。例如后台播放音乐，视频等。
至于其它的方法目前暂时没有发现，如果有新的发现会在此文档后续更新。

ps：个人感觉这个心跳接口跟在线统计的模块功能接口后台设计的不是很合理。由于我没涉及到接口的编写设计，这里不做过多的讨论。

终于有了个人的独立博客

发表于 2017-11-27

胡乱一气，乱说一通

在我学生时代的时候，写作一直是我的诟病。因为我不爱看文学作品，不喜欢写日记，不喜欢动笔写作。但是我却很喜欢数学，物理，喜欢做逻辑题目，典型的理工男。因此，弄一个博客写作放在以前，我肯定是想都不会去想。

但是自从从事研发行业后，我渐渐发现了写作也不是那么无聊的事情。就是一种感觉，觉得写点东西能够抒发自己内心的情绪，能记录自己的成长。于是我开始着手搭建了自己的博客，希望自己能一个月一篇的出货量，一年一次工作成长总结（虽然不知道能不能做到）但是我就是这么想的于是也准备这样做了。搭建博客完成算是我的第一步，虽然没啥技术含量，但是我还是很开心！

博客搭建的技术参考

我博客搭建是采用hexo + github + coding来的方式来做的。
hexo是一款基于Node.js的静态博客框架；
gitbub国外知名项目管理托管平台；
coding国内知名项目托管平台。
主要思路就是用hexo搭建静态本地静态博客后，将其部署到github和coding上面。

具体教程请参考它:Mac上搭建基于GitHub的Hexo博客
